# WebAssembly Deployment

EdgeBot AI supports WebAssembly (WASM) compilation for lightweight, portable AI inference in browser and IoT environments. The `edgebot-wasm` crate provides:

- **Browser target** (`wasm32-unknown-unknown`): Run models directly in web browsers via JavaScript
- **WASI target** (`wasm32-wasi`): Deploy to headless IoT devices with WebAssembly runtime
- **Zero-copy interfaces**: Efficient data passing between JavaScript/Rust and WASM
- **GPU acceleration**: Optional WGPU support for browser-based inference

## Prerequisites

```bash
# Install Rust (if not already)
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

# Add WASM targets
rustup target add wasm32-unknown-unknown wasm32-wasi

# Install wasm-bindgen CLI
cargo install wasm-bindgen-cli

# Install Trunk (for browser builds)
cargo install trunk
```

## Building WASM Modules

The `edgebot-wasm` crate includes a build script `build-wasm.sh` that simplifies compilation for both targets.

### Browser Build

```bash
cd edgebot-wasm
./build-wasm.sh browser
```

Output: `target/wasm/edgebot-wasm-browser.wasm` (requires JavaScript glue code)

### WASI Build

```bash
cd edgebot-wasm
./build-wasm.sh wasi
```

Output: `target/wasm/edgebot-wasm-wasi.wasm` (standalone WASI module)

### Build Options

```bash
# Debug build (unoptimized, with debug symbols)
./build-wasm.sh browser --debug

# Release build with size optimizations
./build-wasm.sh browser --optimize

# Build both targets
./build-wasm.sh all
```

The script uses the following optimization flags:

- `-C opt-level=z` (optimize for size)
- `-C link-arg=-s` (reduce JavaScript glue code)
- `--release` profile
- LTO enabled (`codegen-units = 1`)

## Browser Deployment

### Integrating with JavaScript

After building the browser target, you'll need:

1. The `.wasm` binary file
2. JavaScript glue code (generated by `wasm-bindgen`)

Use the provided `build-dashboard-wasm.sh` script or `trunk` to bundle everything:

```bash
cd edgebot-wasm
./build-wasm.sh browser

# The JS glue code is at:
# target/wasm/edgebot-wasm-browser.js
# target/wasm/edgebot-wasm-browser_bg.wasm (the actual WASM module)
```

### JavaScript API

```javascript
// Import initialization and runtime class
import init, { JsWasmRuntime } from './edgebot-wasm-browser.js';

async function runInference() {
    // Initialize the WASM module
    await init();

    // Create runtime instance
    const runtime = new JsWasmRuntime();

    // Load model (Uint8Array of .ebmodel or .onnx bytes)
    const response = await fetch('models/yolov8.ebmodel');
    const modelBytes = await response.arrayBuffer();
    runtime.load_model('yolo', new Uint8Array(modelBytes));

    // Prepare input tensor (example: 640x640 RGB image)
    const width = 640;
    const height = 640;
    const inputs = [{
        name: 'input',
        data: new Float32Array(width * height * 3), // fill with pixel data
        shape: [1, 3, height, width]
    }];

    // Run inference
    const outputs = runtime.infer('yolo', inputs);

    // Process output
    console.log('Output shape:', outputs[0].shape);
    console.log('Output data:', outputs[0].data);
}

runInference().catch(console.error);
```

### X-Frame-Options and CORS

When serving the WASM files, ensure your server allows cross-origin loading:

```http
Access-Control-Allow-Origin: *
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
```

Or serve from the same origin to avoid CORS issues.

### Performance Tips for Browser

- Use `opt-level = "z"` to minimize WASM binary size
- Enable WGPU backend for GPU acceleration (add `wgpu` feature)
- Cache loaded models in `localStorage` or `IndexedDB` to avoid reloading
- Use `Float32Array` views for zero-copy data passing
- Batch multiple inputs when possible to amortize overhead

## WASI Deployment (IoT)

WASI (WebAssembly System Interface) allows running WASM modules in headless IoT environments like Raspberry Pi, STM32, or cloudflare Workers.

### Running WASI Modules

```bash
# Install WASI runtime (e.g., wasmtime)
curl https://wasmtime.dev/install.sh -sSf | bash

# Run the WASI module
wasmtime target/wasm/edgebot-wasm-wasi.wasm -- \
    --model models/yolov8.ebmodel \
    --input data/input.bin
```

### Rust Host Integration

You can embed the WASM runtime in your Rust IoT application:

```rust
use edgebot_wasm::{WasmRuntime, WasmTarget, WasmInferenceInput};

fn main() -> Result<(), Box<dyn std::error::Error>> {
    // Create WASI runtime
    let mut runtime = WasmRuntime::new(WasmTarget::Wasi);

    // Load model from filesystem
    let model_bytes = std::fs::read("/models/yolov8.ebmodel")?;
    runtime.load_model("yolo", model_bytes, None)?;

    // Prepare sensor input (e.g., camera image)
    let input_data = std::fs::read("/sensor/camera.bin")?;
    let input = WasmInferenceInput {
        name: "input".to_string(),
        data: input_data.chunks(4)
            .map(|chunk| f32::from_le_bytes([chunk[0], chunk[1], chunk[2], chunk[3]]))
            .collect(),
        shape: vec![1, 3, 480, 640],
    };

    // Run inference
    let outputs = runtime.infer("yolo", &[input], None)?;
    let predictions = outputs[0].data;

    // Use predictions for robot control
    process_detections(predictions);

    Ok(())
}
```

### WASI Runtime Options

You can configure the WASI runtime with environment variables:

```bash
# Set WASI memory limits (in pages)
WASI_MEMORY_MAX_PAGES=256 wasmtime ...

# Enable debugging
WASI_DEBUG=1 wasmtime ...
```

### Supported WASI Runtimes

- **wasmtime**: Production-ready, supports WASI preview2
- **wasmer**: Alternative with plugin system
- **wasm3**: Lightweight interpreter for very constrained devices
- **Node.js**: Use `@wasmer/wasi` for Node.js environments

## Cross-compilation for ARM + WASM

EdgeBot CLI can compile models for ARM targets AND produce WASM binaries:

```bash
# Build for ARM (Raspberry Pi) with WASM parallel
edgebot compile --model model.onnx --output ./bin/edgebot-arm \
    --hardware raspberry-pi \
    --wasm-output ./bin/edgebot.wasm

# The resulting WASM binary can be deployed to browser or WASI
```

## Performance Optimization

### Binary Size Reduction

1. Use `opt-level = "z"` (size over speed)
2. Enable LTO (`lto = true`)
3. Set `codegen-units = 1`
4. Strip debug symbols (`strip target.wasm`)
5. Use `wasm-opt` from Binaryen:

```bash
wasm-opt -Oz -o optimized.wasm target.wasm
```

### Memory Usage

- WASM memory defaults to 64KB linear memory; increase with `-C target-feature=+bulk-memory`
- WASI: Use `--max-memory` flag in runtime
- Browser: Memory limits ~2GB (varies by browser)

### Inference Speed

- Prefer `f32` over `f64` (WASM doesn't have native `f64` operations)
- Use SIMD (`target-feature=+simd128`) for vectorized operations if supported
- Keep model tensors in WASM memory to avoid serialization overhead
- Warm up the runtime before measuring (JIT compilation overhead)

## Troubleshooting

| Issue | Solution |
|-------|----------|
| `wasm-bindgen` not found | `cargo install wasm-bindgen-cli` |
| WASM fails to load in browser | Check console for errors; ensure correct MIME type (`application/wasm`) |
| Out of memory in WASI | Increase runtime memory limit (`WASI_MEMORY_MAX_PAGES`) |
| Slow inference | Use `opt-level = "z"`, enable SIMD, consider model quantization |
| `TypeError: WebAssembly.instantiate()` failed | Browser may not support wasm32-unknown-unknown target; use polyfill or update browser |
